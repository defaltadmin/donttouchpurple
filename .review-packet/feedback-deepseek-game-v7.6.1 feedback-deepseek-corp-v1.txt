## DTP Code Review ‚Äî v7.6.1

### [DTP-001] ‚Äî Bomb expiration uses stale pattern, causing visual inconsistency
- **Severity**: Medium  
- **Category**: Correctness  
- **File + Line**: `engine/GameEngine.ts` ‚Üí `_trySpawnBomb` method, lines 861‚Äì884 (specifically the delta timer callback)  
- **Description**: When a bomb is spawned, the `_trySpawnBomb` method captures the current pattern (`pat`) and uses it inside the delta timer callback to call `activeToCellsP(ref.active, pat)`. If the player‚Äôs grid pattern changes (e.g., stage progress, shuffle, or boss event) before the bomb explodes, the old pattern may not match the current grid layout. This can cause `ref.cells` to be incorrectly rebuilt, leading to cells appearing in wrong positions or disappearing.  
- **Impact**: Visual glitches (cells shifting unexpectedly, bombs not clearing correctly, or `ref.cells` becoming inconsistent with `ref.active`). Does not break game logic completely but degrades user experience and can cause confusion.  
- **Fix**: Inside the delta timer callback, retrieve the current pattern from the player‚Äôs state (`const currentPat = EVOLVE_PATTERNS[ref.patternIdx] ?? ...`) instead of using the captured `pat`.  

### [DTP-002] ‚Äî `processTick` lacks disposed guard, can run after engine destruction
- **Severity**: Medium  
- **Category**: Stability  
- **File + Line**: `engine/GameEngine.ts` ‚Üí `processTick` method, line 670  
- **Description**: `processTick` is scheduled via `setTimeout` and can execute even after the engine has been destroyed (e.g., component unmounts while a tick is pending). There is no `if (this._isDisposed) return;` guard at the start of the method, so it may access cleared resources (timers, listeners, player states) and throw exceptions.  
- **Impact**: Potential runtime errors and unhandled exceptions, especially during rapid unmount/remount scenarios (e.g., hot reload, fast navigation). Could cause the game to crash in edge cases.  
- **Fix**: Add `if (this._isDisposed) return;` as the first line of `processTick()`. Also consider adding the guard in `scheduleTick` before scheduling the next tick.

### [DTP-003] ‚Äî Unused `rafIdRef` in `useGameEngine` with misleading cleanup
- **Severity**: Low  
- **Category**: Code Quality  
- **File + Line**: `hooks/useGameEngine.ts` ‚Üí line 57 (`rafIdRef` declaration) and cleanup on line 349  
- **Description**: `rafIdRef` is declared but never assigned a value (the engine‚Äôs RAF is handled inside `GameEngine`, not the hook). The cleanup function attempts to cancel a nonexistent RAF, which is harmless but indicates dead code.  
- **Impact**: No functional impact, but increases technical debt and may confuse maintainers.  
- **Fix**: Remove `rafIdRef` and its associated cleanup block.

### [DTP-004] ‚Äî Missing cleanup for `toastTimeoutRef` in `handleEngineGameOver`
- **Severity**: Low  
- **Category**: Stability  
- **File + Line**: `App.tsx` ‚Üí `handleEngineGameOver` callback, line 610 (the `setTimeout` inside daily objective completion block)  
- **Description**: Inside `handleEngineGameOver`, a timeout is stored in `toastTimeoutRef.current` but there is no corresponding cleanup in a `useEffect` to clear it if the component unmounts before the timeout fires. Other timeouts (e.g., `toastRef`, `comboPopTimerRef`) are cleaned up, but this one is omitted.  
- **Impact**: If the App unmounts while this timeout is pending, the callback would attempt to update state on an unmounted component, causing a React warning and potential memory leak.  
- **Fix**: Add a cleanup effect that clears `toastTimeoutRef.current` on unmount, e.g.:
  ```ts
  useEffect(() => () => { if (toastTimeoutRef.current) clearTimeout(toastTimeoutRef.current); }, []);
  ```

### [DTP-005] ‚Äî Unhandled promise rejection in `challengeLink.parseAndVerify()`
- **Severity**: Low  
- **Category**: Stability  
- **File + Line**: `App.tsx` ‚Üí line 807 (`challengeLink.parseAndVerify().then(...)`)  
- **Description**: The promise returned by `challengeLink.parseAndVerify()` is not caught. If the function rejects (e.g., network error, invalid signature, Firebase unavailable), an unhandled rejection will occur.  
- **Impact**: Silent failure that could be missed in error tracking; may cause the promise rejection to bubble and potentially crash the app if not caught globally.

### [DTP-006] ‚Äî `BuildDeploySection` can be enabled in production (if UI fixed) allowing difficulty cheating
- **Severity**: Low  
- **Category**: Security  
- **File + Line**: `components/Settings/BuildDeploySection.tsx` (entire component) and `App.tsx` line 1678  
- **Description**: `BuildDeploySection` allows runtime modification of difficulty constants via `applyOverride`. Although currently unreachable because the UI button to open it is missing, the component is still imported and can be rendered if `showBuildDeploy` becomes true. In a production environment, a user could theoretically trigger this (e.g., via URL manipulation or devtools) and make the game easier, then submit inflated scores to the global leaderboard.  
- **Impact**: Potential for score cheating if the UI is ever fixed or if an attacker forces `showBuildDeploy` to true. The leaderboard integrity could be compromised.  
- **Fix**: Guard the rendering of `BuildDeploySection` behind a `devMode` flag, e.g.:  
  ```tsx
  {devMode && showBuildDeploy && <BuildDeploySection ... />}
  ```  
  Additionally, ensure `applyOverride` does nothing in non‚Äëdev builds.

### [DTP-007] ‚Äî Firebase API keys exposed to client (expected, but App Check mitigates)

## MSCArabia.com Security & Correctness Review ‚Äî v1.0

### [WEB-001] ‚Äî Bot loop bypasses React state, causing inconsistent UI and memory leaks
- **Severity**: Critical  
- **Category**: Correctness / Memory Leak  
- **File + Line**: `website/src/app/page.tsx` ‚Üí lines 190‚Äì297 (the `setInterval` effect)  
- **Description**: The bot loop uses a mutable `gridRef` to store cell data and directly manipulates DOM elements via `document.querySelector` and `gsap.to`. React state is never updated, so any React re‚Äërender (e.g., parent component re‚Äërender, route change) will reset the grid to `INITIAL_CARDS`, completely discarding all bot progress. Additionally, GSAP tweens created inside the interval are not tracked or killed on unmount, leading to memory leaks and potential errors because they reference stale DOM nodes.  
- **Impact**: The demo grid becomes desynchronised from React‚Äôs virtual DOM. User sees random resets, animations may continue after page navigation, and performance degrades over time.  
- **Fix**: Refactor the bot logic to use React state (`useState<Cell[]>`). Update the state on each bot action and let React re‚Äërender the grid naturally. Remove all manual DOM queries and GSAP tweens inside the interval. If animations are desired, use CSS transitions or React‚Äëbased animation libraries that clean up automatically.

### [WEB-002] ‚Äî GSAP tweens not cleaned up in bot loop (memory leak)
- **Severity**: High  
- **Category**: Memory Leak / Performance  
- **File + Line**: `website/src/app/page.tsx` ‚Üí lines 206‚Äì286 (multiple `gsap.to` and `gsap.fromTo` calls inside interval)  
- **Description**: The bot loop creates numerous GSAP animations (e.g., cell shrink/scale, shake effects) but never stores references to these tweens or kills them on component unmount. GSAP globally retains references to these animations, and they continue to run even after the component is removed, causing CPU usage, memory leaks, and potential ‚ÄúCannot read property of null‚Äù errors.  
- **Impact**: Long‚Äëlived page navigation or hot reloads will accumulate orphaned animations, draining battery and causing console errors.  
- **Fix**: Either avoid GSAP inside the bot loop entirely (use CSS transitions or React state‚Äëbased re‚Äërenders), or store each tween in a ref and call `tween.kill()` on cleanup. Alternatively, use `gsap.globalTimeline.clear()` in the effect cleanup, though this is heavy. Preferred fix: replace bot loop with state updates and remove GSAP from the interval.

### [WEB-003] ‚Äî Missing Content Security Policy (CSP) headers
- **Severity**: Medium  
- **Category**: Security  
- **File + Line**: `website/next.config.ts` (no CSP) and `website/src/app/layout.tsx` (no meta CSP)  
- **Description**: The application is exported as static HTML with no Content Security Policy headers. While the site does not accept user input, a compromised third‚Äëparty script (e.g., Google Fonts CDN) could inject malicious code. CSP would restrict script sources and mitigate XSS risks.  
- **Impact**: Potential for script injection via external resources (fonts, analytics) or if any user‚Äëcontrolled data is ever introduced.  
- **Fix**: Add a `<meta http-equiv="Content-Security-Policy" ...>` tag in `layout.tsx` with a reasonable default policy (e.g., `default-src 'self'; script-src 'self' 'unsafe-inline' https://cdnjs.cloudflare.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src https://fonts.gstatic.com; connect-src 'self'; img-src 'self' data:;`). For production, consider generating the policy via Next.js headers in a custom server or via middleware.

### [WEB-004] ‚Äî `NebulaCanvas` missing `renderer.destroy()` ‚Äì WebGL context leak
- **Severity**: Medium  
- **Category**: Memory Leak  
- **File + Line**: `website/src/components/NebulaCanvas.tsx` ‚Üí cleanup function, line 169  
- **Description**: The OGL `Renderer` instance is created but never destroyed. The cleanup only calls `gl.getExtension('WEBGL_lose_context')?.loseContext()`, which forces a context loss but does not free all GPU resources allocated by OGL (e.g., programs, geometries). OGL provides a `renderer.destroy()` method that properly cleans up all WebGL objects.  
- **Impact**: Repeated mounting/unmounting of the component (e.g., during development or if the landing page is shown/hidden) will leak GPU memory, potentially leading to tab crashes or performance degradation.  
- **Fix**: In the cleanup function, call `renderer.destroy()` before losing the context:
  ```ts
  return () => {
    cancelAnimationFrame(rafId);
    window.removeEventListener('resize', resize);
    window.removeEventListener('mousemove', onMouseMove);
    renderer.destroy();
    gl.getExtension('WEBGL_lose_context')?.loseContext();
  };
  ```

### [WEB-005] ‚Äî Hydration mismatch on footer copyright year
- **Severity**: Low  
- **Category**: UX / Correctness  
- **File + Line**: `website/src/app/page.tsx` ‚Üí line 421 (`<footer>{new Date().getFullYear()}...`)  
- **Description**: Because the component is marked `'use client'`, Next.js still pre‚Äërenders it on the server at build time (static export). The server renders the year as the build year, while the client will render the current year after hydration, causing a mismatch. React will discard the server HTML and re‚Äërender on the client, which is harmless but produces a console warning and a visible flicker.  
- **Impact**: Minor hydration warning; does not break functionality but indicates a hydration anti‚Äëpattern.  
- **Fix**: Use `useState` and `useEffect` to set the year only on the client:
  ```tsx
  const [year, setYear] = useState<number | null>(null);
  useEffect(() => setYear(new Date().getFullYear()), []);
  return <footer>&copy; {year ?? '2026'} Don't Touch Purple ...</footer>;
  ```

### [WEB-006] ‚Äî `setInterval` runs when tab is hidden (battery drain)
- **Severity**: Low  
- **Category**: Performance  
- **File + Line**: `website/src/app/page.tsx` ‚Üí line 190 (`setInterval` effect)  
- **Description**: The bot loop uses `setInterval` that runs every 600ms regardless of whether the page is visible. When the tab is backgrounded, the interval continues to execute, wasting CPU and battery.

### [WEB-008] ‚Äî `document.querySelector` inside GSAP callbacks may target stale elements
- **Severity**: Medium  
- **Category**: Stability  
- **File + Line**: `website/src/app/page.tsx` ‚Üí lines 209, 236, 262, etc. (multiple `document.querySelector(`.cell-${idx}`)`)  
- **Description**: The bot loop uses `document.querySelector` to find DOM elements inside GSAP `onComplete` callbacks. If the component re‚Äërenders (e.g., due to a state change elsewhere) between the time the animation starts and the callback executes, the queried element may have been removed from the DOM or replaced by a React re‚Äërender, leading to a `null` reference and potential errors.  
- **Impact**: Intermittent crashes or silent failures, especially if the page ever has any state changes (which currently it does not, but future maintenance could introduce them).  
- **Fix**: Avoid manual DOM queries inside React components. Instead, use React refs to access elements, or (preferably) use React state updates to re‚Äërender the grid rather than directly manipulating DOM styles. If animations are needed, use React‚Äëbased animation libraries like `framer-motion` that integrate with the component lifecycle.
